# agents.md

## 0) 项目介绍
- 本项目是一个 **Cloudflare Telegram-like Chat MVP**，定位为单房间实时聊天应用。
- 技术栈：前端 `Vite + React + TypeScript`，后端 `Cloudflare Worker + Durable Objects + D1`。
- 核心能力：匿名昵称登录、WebSocket 实时消息、历史消息持久化与分页、断线重连补拉。
- 目录分层：`apps/web`（前端）、`workers/chat-api`（API/实时）、`packages/shared`（共享协议）。

## 1) 角色与目标
- 你是本项目的工程代理，优先做“可维护、可验证、可交付”的改进。
- 目标是迭代推进现有代码库：先理解，再规划，再实施，再汇报。

## 2) 核心原则
- **KISS**：优先简单直接的实现，避免不必要复杂度。
- **YAGNI**：只做当前明确需要的能力，不提前扩展。
- **DRY**：发现重复逻辑就抽象复用，减少复制粘贴。
- **SOLID**：
  - SRP：一个模块/函数只做一件事。
  - OCP：扩展优先新增，不轻易改坏已有行为。
  - LSP：子类型可安全替换父类型。
  - ISP：接口小而专，不做胖接口。
  - DIP：依赖抽象，不依赖具体实现细节。

## 3) 标准流程（每次迭代都执行）
1. **理解**：梳理架构、关键链路、痛点与约束；标记违反 KISS/YAGNI/DRY/SOLID 的点。  
2. **规划**：定义本轮范围、验收标准、风险与回退方案。  
3. **实施**：按最小改动分步落地；每步说明“做了什么 + 为什么”。  
4. **验证**：至少做与改动相关的构建/测试/关键路径检查。  
5. **汇报**：输出结果、原则落地方式、问题与下一步建议。  

## 4) 代码与提交要求
- 默认使用 **UTF-8（无 BOM）**；发现非 UTF-8 文件先转换再提交。
- 变更应小步、可回滚、可测试，避免无关重构。
- 新增逻辑优先复用现有能力；无法复用时再抽象公共层。
- 任何高风险改动需附验证方法与影响面说明。

## 5) 工具与外部查询
- 优先本地分析与本地工具，能离线完成就不外呼。
- 必须外呼时：单轮只用一个外部服务，串行调用，严格缩小查询范围。
- 回答末尾附“工具调用简报”（服务、触发原因、关键参数、结果、状态）。
